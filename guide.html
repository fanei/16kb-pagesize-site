16KB Pagesize Optimization Guide

This comprehensive guide walks you through achieving sub-16KB first-load payloads for instant page rendering.

Understanding the 16KB Budget

The 16KB limit refers to the compressed size of critical resources (HTML + critical CSS + essential JS) that must load in the first network round trip. Here's how we allocate it:

Resource Type Budget Purpose
HTML Structure 2-3KB Semantic markup, essential content
Critical CSS (inlined) 2-4KB Layout, typography, above-the-fold styles
Essential JavaScript 1-2KB Basic interactions, smooth scroll
Buffer 7-11KB Room for content expansion
üí° Tip: After Gzip compression, our 7.5KB HTML becomes ~2.5KB, leaving significant room for growth while maintaining instant load times.

Step 1: Inline Critical CSS

Why Inline CSS?

External CSS files create render-blocking requests. Inlining critical CSS eliminates these requests and allows immediate rendering.

How to Do It:

  1. Identify critical CSS: Styles needed for above-the-fold content (header, navigation, first section)
  2. Extract and minify: Remove whitespace, shorten selectors, combine rules
  3. Embed in <head>: Place minified CSS between <style> tags
<head>
    <style>
        *{margin:0;padding:0;box-sizing:border-box}
        body{font-family:-apple-system,BlinkMacSystemFont,"Segoe UI",Roboto,sans-serif;line-height:1.6;color:#212121}
        .container{max-width:1200px;margin:0 auto;padding:0 16px}
        /* ... more critical styles ... */
    </style>
</head>

Step 2: Defer Non-Critical JavaScript

Strategy:

  • Use async or defer attributes for external scripts
  • Keep essential JS minimal (smooth scroll, basic interactions)
  • Load heavy libraries only when needed
<!-- Defer non-critical scripts -->
<script defer src="analytics.js"></script>

<!-- Inline essential JS -->
<script>
    // Smooth scroll, basic interactions
    document.querySelectorAll('a[href^="#"]').forEach(anchor => {
        anchor.addEventListener('click', function (e) {
            // ... smooth scroll logic
        });
    });
</script>

Step 3: Optimize HTML Structure

Techniques:

  • Remove unnecessary whitespace: Minify HTML (but keep readability during development)
  • Use semantic HTML: Reduces need for extra CSS classes
  • Avoid inline styles: Keep styles in CSS for better compression
  • Minimize comments: Remove HTML comments in production

Step 4: Enable Compression

Server-Side Compression:

Most modern hosting platforms (Netlify, Vercel, Cloudflare) automatically enable Gzip/Brotli compression. This typically reduces file size by 60-70%.

Verify Compression:

  1. Open Chrome DevTools ‚Üí Network tab
  2. Refresh page
  3. Check "Size" vs "Transferred" columns
  4. If "Transferred" is much smaller, compression is working

Step 5: Performance Measurement

Key Metrics:

  • First Contentful Paint (FCP): < 1.0s (excellent), < 2.5s (good)
  • Largest Contentful Paint (LCP): < 2.5s (excellent), < 4.0s (good)
  • Time to Interactive (TTI): < 3.0s (excellent)
  • Total Blocking Time (TBT): < 200ms (excellent)

Tools:

  • Chrome DevTools Lighthouse: F12 ‚Üí Lighthouse tab ‚Üí Generate report
  • Google PageSpeed Insights: https://pagespeed.web.dev/
  • WebPageTest: https://www.webpagetest.org/

Flutter Optimization Tips

Reduce App Size:

  • Use flutter build apk --split-per-abi to create separate APKs per architecture
  • Enable R8/ProGuard for code shrinking and obfuscation
  • Remove unused assets and fonts
  • Use vector graphics instead of bitmaps when possible

Video Player Optimization:

  • Choose lightweight packages (td_fplayer, better_player)
  • Enable hardware acceleration
  • Use adaptive bitrate streaming
  • Lazy load video players (only load when needed)

Common Pitfalls to Avoid

‚ùå Don't:

  • Load entire CSS frameworks (Bootstrap, Tailwind) in critical path
  • Use large JavaScript libraries for simple tasks
  • Include images in critical CSS (use data URIs sparingly)
  • Block rendering with synchronous scripts
  • Forget to enable compression on server

‚úÖ Do:

  • Measure before and after optimization
  • Test on slow 3G networks
  • Use progressive enhancement
  • Keep critical path minimal
  • Monitor Core Web Vitals
